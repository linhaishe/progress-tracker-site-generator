# Progress Tracker Site Generator Design

## Purpose

Create a focused npm package repo for `@linhaishe/progress-tracker-site`.

The package installs a read-only static progress dashboard into a project that uses the `context/*.md` tracker convention. The primary command is:

```bash
npx @linhaishe/progress-tracker-site init
```

The generated page helps the next agent or developer answer two questions quickly:

- What is the current project state?
- What should happen next?

## Repository Direction

This repository should be treated as the progress tracker site generator package, not as a full AI Project Kit monorepo.

Existing historical specs are source material only:

- `docs/superpowers/specs/2026-06-28-progress-tracker-visualization-design.md` describes a superseded Next.js `/dev/progress` route and should not drive implementation.
- `docs/superpowers/specs/2026-06-28-progress-tracker-site-kit-design.md` contains useful static-site concepts, but its kickoff-skill and kit-repository scope should not be carried into this repo's v1.

## V1 Scope

V1 includes:

- CLI command: `init`.
- Package name: `@linhaishe/progress-tracker-site`.
- Static HTML generation.
- Tracker skeleton creation when files are missing.
- Conservative conflict handling.
- A polished generic dashboard layout based on the bundled design template.
- Manual in-browser refresh by pasting or loading updated markdown.

V1 does not include:

- A Next.js route.
- A server, API, database, auth, or deployment workflow.
- Editing tracker markdown from the generated page.
- Automatic synchronization between progress and milestone trackers.
- GitHub issue, PR, or project-management integrations.
- Kickoff skill implementation.

## Inputs

The generator works with these files in the target project:

- `context/progress-tracker.md`
  - Handoff state: current phase, current goal, next workflow, next up, open questions, risks, assumptions, decisions, session notes, and verification.
- `context/milestone-tracker.md`
  - Lifecycle state: current stage, MVP readiness, release confidence, milestones, current milestone exit criteria, MVP scope, blockers, and milestone notes.

The visual system comes from the package's bundled template:

- `templates/context/design.md`

The target project does not need to provide its own `context/design.md` for v1.

## Design Template Policy

`templates/context/design.md` should become a generic progress dashboard design template.

Keep:

- Warm paper-like background and surfaces.
- Dense operational dashboard layout.
- Crisp borders with minimal shadows.
- Amber accents used sparingly for orientation.
- Arial/Helvetica typography scale.
- 4px/8px spacing rhythm.
- Radius values at `4px`, `6px`, and `8px`.
- Responsive stacking rules that avoid horizontal overflow.

Remove or rewrite:

- Product-specific names.
- Customer-service-specific rules.
- AI support workflow language.
- Reference-site evidence sections.
- Any wording that implies the generated page is tied to a specific application domain.

The generator should not attempt complex natural-language parsing of this design file in v1. The implementation should translate the bundled generic design into stable static CSS.

## Output

Default output path:

- If the target project has a `docs/` directory, generate `docs/progress-tracker.html`.
- Otherwise generate `progress-tracker.html` at the project root.

The user may override this with:

```bash
npx @linhaishe/progress-tracker-site init --output docs/progress-tracker.html
```

The generated HTML must be:

- Self-contained.
- Openable directly in a browser.
- Read-only.
- Usable without build tooling.
- Marked as generated by this package.

## CLI Interface

Primary command:

```bash
npx @linhaishe/progress-tracker-site init
```

Supported flags:

```bash
npx @linhaishe/progress-tracker-site init --root .
npx @linhaishe/progress-tracker-site init --output docs/progress-tracker.html
npx @linhaishe/progress-tracker-site init --dry-run
npx @linhaishe/progress-tracker-site init --force
```

Expected output should report:

- Target root.
- Detected output path.
- Created files.
- Existing files preserved.
- Conflicts skipped.
- How to open the generated HTML file.

## File Creation And Conflict Rules

Default behavior is conservative:

- Create `context/progress-tracker.md` if missing.
- Create `context/milestone-tracker.md` if missing.
- Preserve existing tracker markdown files.
- Generate the HTML page when no output file exists.
- Preserve an existing HTML file if it does not contain this tool's generated marker.
- Allow overwriting a previously generated HTML file with `--force`.
- In `--dry-run`, print the intended plan without writing files.

Generated HTML should include a clear marker such as:

```html
<!-- Generated by @linhaishe/progress-tracker-site. -->
```

## Generated Page Information Architecture

The generated page should use the approved Handoff Cockpit layout.

First screen:

- Header
  - Title: `Project Progress`
  - Source file summary.
  - Generated timestamp.
  - Read-only badge.
  - Manual refresh affordance.
- Left primary column
  - Current phase or stage.
  - Current goal.
  - Recommended next workflow.
  - Signal cards for next up, open questions, and verification.
- Right rail
  - Milestone status.
  - Current milestone exit criteria.
  - Risks and blockers.
  - Assumptions.
  - Refresh source controls.

Below the first screen:

- Completed work.
- In-progress work.
- Architecture decisions.
- Governance or review notes when present.
- Session notes.
- Full milestone details.
- Unknown sections preserved as additional notes.

## Manual Refresh Behavior

The page embeds a snapshot of tracker data at generation time.

The page also provides browser-only controls to update the rendered view without writing files:

- Paste updated `progress-tracker.md`.
- Paste updated `milestone-tracker.md`.
- Load local markdown files through browser file inputs.
- Reset to the embedded snapshot.

These controls update only the in-memory rendered page. They do not write back to disk.

## Parsing Strategy

Use a small parser built around the tracker convention rather than a full Markdown renderer.

For both trackers:

- Parse `#` title.
- Split sections by level-two headings (`##`).
- Preserve known sections in named fields.
- Preserve unknown sections in an additional notes area.
- Parse checklist counts from `- [x]` and `- [ ]` where useful.
- Preserve plain text lines and bullet lists.

Missing or empty sections are visible state, not silent omissions.

Examples:

- Missing tracker file: create skeleton during install.
- Missing critical section in an existing tracker: show a quiet warning in the generated page.
- Empty known section: show a neutral empty state.
- Unknown section: render under additional notes.

## Module Boundaries

Recommended package structure:

```text
package.json
bin/
  progress-tracker-site.js
src/
  cli.ts
  commands/
    init.ts
  core/
    file-plan.ts
    output-path.ts
    ownership.ts
  trackers/
    parse-progress-tracker.ts
    parse-milestone-tracker.ts
    skeletons.ts
  render/
    render-page.ts
    static-assets.ts
templates/
  context/
    design.md
    progress-tracker.md
    milestone-tracker.md
  static-progress-page/
    README.md
```

Boundary rules:

- CLI modules handle arguments and reporting.
- File-plan modules decide what to create, preserve, skip, or overwrite.
- Tracker modules parse markdown and provide skeleton content.
- Render modules produce the final self-contained HTML.
- Templates provide default content and design direction.

## Error Handling

CLI errors should be direct and recoverable:

- Invalid root: explain the missing path.
- Invalid output path: explain whether parent directories can be created.
- Conflict: name the file and explain `--force` only works for generated HTML.
- Unexpected read/write failure: show the path and system error message.

The generated page should avoid stack traces. It should show human-readable warnings for missing sections or invalid pasted content.

## Testing And Verification

Implementation should include focused tests for:

- Output path detection.
- Dry-run file planning.
- Conflict handling for generated and non-generated HTML.
- Skeleton creation.
- Progress tracker parsing.
- Milestone tracker parsing.
- Checklist counting.
- HTML marker insertion.
- Basic generated HTML content.

Manual verification should include:

- `npm run build` or equivalent package build.
- CLI dry run in a temporary project.
- CLI install in a temporary project.
- Opening generated HTML directly in a browser.
- Pasting updated progress markdown.
- Loading local markdown files.
- Mobile-width visual check for no horizontal overflow.

## Success Criteria

V1 is successful when:

- `npx @linhaishe/progress-tracker-site init` installs a usable static dashboard in a project.
- Existing tracker markdown is preserved.
- Missing tracker markdown receives generic skeletons.
- Existing non-generated HTML is not overwritten by default.
- The generated page clearly communicates current handoff state and milestone progress.
- The page can be refreshed manually in the browser without writing files.
- The visual design matches the cleaned generic dashboard template.
