V1 Direction: Generator + Static Dashboard

Pick the behavior level you want for the generated HTML page. All options use `context/progress-tracker.md`, `context/milestone-tracker.md`, and `context/design.md` as inputs.

A. Snapshot Page
Project Progress | Generated once
Current Goal
Milestone Status
Read-only sections rendered at generation time

Simple Snapshot

CLI reads markdown and writes a self-contained HTML snapshot. Lowest risk and easiest to publish, but users rerun the CLI whenever tracker files change.

B. Snapshot + Paste/Load Refresh
Project Progress | Static + refresh controls
Handoff cockpit from embedded snapshot
Paste updated markdown
Milestones, blockers, next workflow

Recommended: Static Dashboard with Manual Refresh

Generates a polished embedded snapshot, plus client-side controls to paste or load updated tracker markdown. Still no server, no build, no writes.

C. Local Reader Tool
Project Progress | Browser app mode
Sections
Filters
Source inputs
Interactive parsing, tabs, raw source view, richer controls

Interactive Local Reader

More app-like static HTML with tabs, filters, raw markdown panes, and richer controls. Useful, but bigger v1 surface and more test burden.

My recommendation

Choose B. It keeps the package honest as a static generator while making the page useful after `progress-tracker.md` changes, without inventing a server or syncing model.
